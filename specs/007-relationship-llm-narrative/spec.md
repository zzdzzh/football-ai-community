# Feature Specification: 关系分析 LLM 叙事解读

**Feature Branch**: `007-relationship-llm-narrative`

**Created**: 2026-07-17

**Status**: Draft

**Input**: User description: "关系 LLM 叙事 —— 在已交付的球员关系分析（005）结构化结论之上，用大模型生成可读的中文关系介绍；仅基于已入库履历与计算出的关系结论，禁止捏造；失败时降级为结构化面板"

**Parent Vision**: [000-football-community-vision](../000-football-community-vision/spec.md)

**Prerequisites**: [005 球员关系分析](../005-player-relationship-analysis/spec.md) 已可产出就绪的球员对分析结论；[001 认证](../001-football-feed-mvp/spec.md) 登录策略与关系页一致。本 feature 兑现 005 **FR-019** 预留边界，并解除 005 **FR-018** 对本能力的「本 MVP 不交付」限制（仅限本 feature 范围，不改写 005 履历计算本身）。

## Clarifications

- **当前无待澄清问题**：交付形态默认为「一次性关系叙事介绍」（非多轮自由问答）；触发方式为分析就绪后由用户主动请求生成；超时/失败降级为既有结构化结论面板；纳入既有 AI 提问限流；语言为简体中文。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 基于已入库结论生成关系叙事 (Priority: P1)

作为登录用户，当两名球员的关系分析已得出可验证的结构化结论后，我希望系统用自然语言写一段中文关系介绍，把队友、转会关联、间接路径等结论讲清楚，且介绍内容不得超出已入库履历与本次分析结论。

**Why this priority**: 这是相对结构化面板的差异化体验，也是 005 明确延后、006 声明另开的核心能力；无此则仍停留在「表格结论」。

**Independent Test**: 对一对分析状态为就绪、且含至少一项明确关系结论（如俱乐部队友成立）的球员对请求叙事；返回一段中文介绍，且可人工核对其中事实主张均能在结构化结论中找到依据；不出现结论中不存在的共同效力或比赛经历。

**Acceptance Scenarios**:

1. **Given** 球员对分析已就绪且含结构化结论，**When** 登录用户请求生成关系叙事，**Then** 系统返回一段简体中文介绍，覆盖结论中的主要关系要点（如队友/转会/间接路径中已成立或已给出的项）
2. **Given** 结构化结论中某关系为「不成立」或「无法判定」，**When** 生成叙事，**Then** 叙事不得改写为「成立」或编造共同效力；「无法判定」须以不确定语气保留
3. **Given** 结构化结论中无任何共同效力、转会关联或间接路径，**When** 生成叙事，**Then** 介绍须如实说明「当前数据未建立可验证关联」，不得捏造路径或共同经历

---

### User Story 2 - 关系页展示叙事与失败降级 (Priority: P2)

作为登录用户，我在关系分析页查看结构化时间线/关系图的同时，希望能看到（或一键生成）AI 关系叙事；若生成超时、限流或服务失败，页面仍完整展示原有结构化结论，不得整页空白或误导为「无关系」。

**Why this priority**: 用户可见闭环；降级策略直接对应 005 FR-019，也是上线后最常见的故障路径。

**Independent Test**: 打开已就绪分析页，触发叙事生成并看到介绍区域；模拟失败/超时后，结构化面板仍可见且有明确失败提示。

**Acceptance Scenarios**:

1. **Given** 用户已登录且分析页结论就绪，**When** 用户触发「生成关系解读」，**Then** 页面在加载态后展示叙事正文，并标明内容由 AI 基于本页结论生成
2. **Given** 叙事生成超时或上游失败，**When** 结果返回，**Then** 页面保留完整结构化关系结论，并显示可读的失败/超时提示（非静默空白）
3. **Given** 用户触发频率超过平台 AI 限流策略，**When** 再次请求叙事，**Then** 提示需稍后重试，且不影响已展示的结构化结论

---

### User Story 3 - 叙事可重入与按结论版本复用 (Priority: P3)

作为登录用户，我希望对同一球员对、在分析结论未变化时再次打开页面，能看到已生成过的叙事（或快速再次获取同一版本），避免每次进入都重新等待；若用户强制刷新分析导致结论更新，则可重新生成叙事。

**Why this priority**: 降低重复调用成本、提升重入体验；可独立验收，但非首屏最小闭环所必需。

**Independent Test**: 对同一就绪球员对连续两次进入分析页并请求叙事；第二次可复用已有叙事或在秒级内返回同等内容；强制重算分析后允许生成新叙事。

**Acceptance Scenarios**:

1. **Given** 某球员对已成功生成过叙事且分析结论未变，**When** 用户再次进入该分析页并查看叙事，**Then** 可看到已保存的叙事，无需必然重新等待完整生成
2. **Given** 用户触发分析强制重算且新结论已就绪，**When** 再次请求叙事，**Then** 系统可基于新结论生成更新后的叙事（或明确提示需重新生成）
3. **Given** 用户未登录，**When** 尝试请求叙事，**Then** 被拒绝并引导登录（与关系分析页鉴权一致）

---

### Edge Cases

- 分析仍在计算中（非就绪）：不得生成叙事；提示等待分析完成
- 履历同步失败或部分缺失导致多项「无法判定」：叙事须如实反映不确定，不得补全为确定事实
- 间接路径为空（上限内无路径）：叙事不得编造中间球星或俱乐部节点
- 「共同夺冠」等荣誉类主张：叙事 MUST NOT 当作已验证事实输出（与 005 边界一致）
- 上游大模型返回与输入结论矛盾的主张：系统 MUST 拒绝采信矛盾内容或降级为结构化面板，不得原样展示无法核验的捏造段落
- 网络中断或短时抖动：展示重试入口；结构化面板始终可用

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 仅在球员对分析结论已就绪时允许生成关系叙事
- **FR-002**: 叙事生成的输入 MUST 仅包含已入库履历摘要与本功能（005）已计算出的结构化关系结论；MUST NOT 引入未入库的外部实时传闻作为事实依据
- **FR-003**: 叙事 MUST NOT 捏造共同效力时段、共同出场、同场对决、荣誉或结论中不存在的间接路径节点
- **FR-004**: 当结论含「无法判定」或「不成立」时，叙事 MUST 保持与结论一致的语义，不得升级为「成立」
- **FR-005**: 系统 MUST 在超时、上游失败或内容核验失败时降级：用户仍可完整查看结构化关系结论面板
- **FR-006**: 关系分析页 MUST 提供叙事展示区域与明确的生成/加载/失败反馈；失败时 MUST NOT 静默空白
- **FR-007**: 叙事相关请求 MUST 要求用户已登录；未登录 MUST 拒绝并引导登录（与关系页一致）
- **FR-008**: 叙事生成 MUST 纳入既有「同用户 × AI 能力」提问限流策略；超限时返回可理解的稍后重试提示
- **FR-009**: 系统 MUST 将生成所用提示与模型调用配置外置（不硬编码在业务分支中随意改写事实规则）
- **FR-010**: 对同一球员对、同一分析结论版本，系统 MUST 支持叙事结果的持久化或等价复用，以便页面重入时不必每次强制重新生成
- **FR-011**: 叙事展示 MUST 向用户标明「由 AI 基于本页结构化结论生成」，避免被误认为原始数据源原文
- **FR-012**: 本 feature MUST NOT 修改 005 的履历采集规则、队友/路径计算语义，也 MUST NOT 静默改写 005/006 既有契约字段含义；若需扩展 005 分析响应以挂载叙事，须在对本模块契约中声明，并对 005 的 `tasks.md` 按多模块规约增补协作任务（如有）
- **FR-013**: MVP MUST 交付「一次性关系叙事介绍」；MUST NOT 将多轮自由问答式「关系聊天」纳入本期范围

### Key Entities *(include if feature involves data)*

- **球员对分析结论（Pair Analysis Result）**：005 已计算并持久化的结构化结论（队友、转会关联、间接路径、时间线/图数据等）；本 feature 的唯一事实来源
- **关系叙事（Relationship Narrative）**：针对某一球员对、绑定某一分析结论版本的一段自然语言介绍；含正文、生成时间、与结论的关联标识、可选状态（就绪/失败）
- **叙事生成请求（Narrative Generation Request）**：登录用户针对指定球员对发起的一次生成或复用读取动作

### 术语定义

| 术语 | 定义 |
|------|------|
| **分析就绪** | 球员对分析已完成计算并可供展示结构化结论（非计算中、非失败态） |
| **关系叙事** | 基于结构化结论生成的一次性自然语言介绍正文 |
| **结构化结论面板** | 005 已交付的文字结论、时间线、关系图等非 LLM 展示 |
| **降级** | 叙事不可用时仍完整展示结构化结论面板，并给出失败/超时/限流说明 |
| **结论版本** | 某次分析计算产出的结论快照标识；结论变化则视为新版本 |
| **事实主张** | 叙事中可被核对的陈述（如「曾为某俱乐部队友」）；须能在结构化结论中找到依据 |

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001** [Deferrable: no]: 在约定样例集（≥5 对含明确关系、≥3 对无明确关联）上，人工抽检叙事中的事实主张与结构化结论 **零矛盾**（无捏造共同效力/路径/荣誉）
- **SC-002** [Deferrable: no]: 登录用户在分析已就绪的页面上，**60 秒内**可完成「触发生成 → 看到叙事正文」或「看到明确失败提示且结构化面板仍完整」二者之一
- **SC-003** [Deferrable: no]: 故意制造超时/失败时，**100%** 样例仍可阅读完整结构化结论，且失败提示可见
- **SC-004** [Deferrable: yes]: 对分析结论未变化的同一球员对，第二次进入页面获取叙事时，**80%** 的请求在 **3 秒内**展示已有叙事（复用路径；可后验）

## 范围边界

### In Scope

- 基于就绪球员对分析结论的一次性中文关系叙事生成与展示
- 失败/超时/限流降级到结构化面板
- 叙事与结论版本的关联及重入复用
- 登录鉴权与 AI 限流纳入
- 提示/模型调用配置外置与「AI 生成」标识

### Out of Scope

- 多轮自由问答、追问「他们谁更强」等开放对话 Agent
- 基于未入库新闻/社交媒体的关系八卦采信
- 修改 005 履历同步、队友判定、BFS 路径算法本身
- 管理员审核叙事内容的工作流
- 付费、计费、实时语音播报
- 自动把叙事同步进 Feed 作为独立内容类型（可后续 feature）

## Assumptions

- 用户已能通过 005 完成搜索消歧并进入 `/relationships/:playerIdA/:playerIdB`，且分析可到达就绪态
- 「叙事」指一段可读介绍（约数百字量级即可），不是百科全书或比赛逐事件解说
- 语言默认简体中文；暂不要求多语言
- 生成触发以用户主动请求为主，避免分析一就绪就自动消耗配额（利于限流与成本）
- 平台已有可用的 OpenAI 兼容模型配置；本 feature 不引入第二套模型供应商
- 006 身份映射不是叙事生成的前置条件；叙事仅消费履历域关系结论
