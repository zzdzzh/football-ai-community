**Updated**: 2026-07-15

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 双球员搜索与直接关系分析 (Priority: P1)

作为球迷，我希望输入两名球员姓名并确认人选后，系统自动判断他们是否曾是**俱乐部队友**或**国家队队友**（效力时间存在交集），并展示共同球队/国家队与共同效力时段，以便快速理解两人的直接关联。

**Why this priority**: 直接队友关系是本功能的核心价值与可独立交付的 MVP 闭环；无此能力则后续路径分析与可视化无意义。

**Independent Test**: 在本地已具备梅西与苏亚雷斯履历数据的前提下，搜索并选定两人后，可在 10 秒内看到「曾经是俱乐部队友」、共同球队含 FC Barcelona，以及可验证的共同效力时间区间；若仅有国家队重叠则可验证国家队结论。

**Acceptance Scenarios**:

1. **Given** 用户进入球员关系搜索页并输入两个可唯一定位的球员姓名，**When** 分别选定候选结果并提交分析，**Then** 进入携带两名球员核心 ID 的关系分析页，并展示两类直接关系结论（俱乐部队友 / 国家队队友）各自为「成立」或「不成立」
2. **Given** 两名球员曾效力同一俱乐部且效力时间存在交集，**When** 完成分析，**Then** 俱乐部关系结果显示「曾经是队友」、列出全部符合条件的共同俱乐部名称及对应共同效力起止时间
3. **Given** 两名球员代表过同一国家队且国家队效力时间存在交集，**When** 完成分析，**Then** 国家队关系结果显示「曾经是国家队队友」、国家队名称与共同效力起止时间
4. **Given** 用户输入的姓名匹配到多名球员，**When** 搜索返回候选，**Then** 系统要求用户从候选列表中明确选择，不得静默任选一人进行分析
5. **Given** 任一名球员履历尚不可用且外部数据源暂时失败，**When** 用户提交分析，**Then** 页面展示友好失败提示与可重试入口，不得伪造履历或关系结论

---

### User Story 2 - 转会关联与间接关系路径 (Priority: P2)

作为球迷，我希望在直接队友分析之外，进一步看到两人是否存在**直接转会关联**、是否先后加入同一球队，以及最短的**间接关系路径**与关系距离，以便理解「非同时队友」的连接方式。

**Why this priority**: 扩展直接关系之外的探索价值；可在 US1 成立后增量交付，且不阻塞核心直接关系闭环。

**Independent Test**: 选定两名无共同效力交集、但可通过共同俱乐部节点连通的球员后，可看到非空的间接关系路径、数值型关系距离，以及「是否先后加入同一球队」等转会关联结论（成立或不成立及依据摘要）。

**Acceptance Scenarios**:

1. **Given** 两名球员均有完整俱乐部履历，**When** 打开关系分析页，**Then** 系统给出转会关系结论：是否存在可验证的直接转会关联、是否存在「先后效力同一俱乐部」情形，并附简短依据（俱乐部名与先后时间）
2. **Given** 两名球员在球员—俱乐部关系网络中可连通，且最短路径跳数不超过产品约定上限，**When** 完成间接关系分析，**Then** 展示一条最短关系路径（交替出现球员与俱乐部节点）及关系距离（跳数）
3. **Given** 在约定跳数上限内不存在连通路径，**When** 完成间接关系分析，**Then** 明确展示「在当前数据与距离上限内无间接关系」，不得编造路径
4. **Given** 用户刷新或通过带球员 ID 的 URL 重新进入分析页，**When** 页面加载，**Then** 仍展示针对同一对球员的同一套分析结果（可重入）

---

### User Story 3 - 关系时间线与关系图可视化 (Priority: P3)

作为球迷，我希望在关系分析页以**时间线**查看两人履历在时间轴上的对齐情况，并以**关系图**查看球员、球队节点及连接路径，以便直观理解共同效力时段与间接关联。

**Why this priority**: 可视化提升理解效率，但依赖 US1/US2 的分析结果；可独立验收展示正确性，不阻塞文本结论交付。

**Independent Test**: 对已有俱乐部队友关系的球员对，时间线可标出共同效力年份/区间，关系图中可见两名球员节点与至少一个共同俱乐部节点及边；无关系时图与时间线不出现虚假共同节点。

**Acceptance Scenarios**:

1. **Given** 分析结论中存在共同俱乐部效力交集，**When** 查看时间线区域，**Then** 按年份或时间段展示两人对该俱乐部的效力条目，并突出共同效力区间
2. **Given** 分析结论中存在间接或直接关系路径，**When** 查看关系图区域，**Then** 图中至少包含路径上的球员节点与俱乐部节点，边与文字结论路径一致
3. **Given** 直接与间接关系均不成立，**When** 查看可视化区域，**Then** 展示空状态说明，时间线仍可分别展示两人各自履历（若数据可用），但不得暗示虚假共同关系
4. **Given** 关系分析页加载中或外部数据仍在补充，**When** 用户停留在页面，**Then** 展示明确的加载/同步状态，完成后自动刷新结论与可视化，或提示用户重试

---

### Edge Cases

- **重名球员**：必须列出可区分线索（如出生年份、主要俱乐部）供用户选择；未选定前禁止进行分析
- **履历时间缺失或不完整**（仅有加入年无离队日、或仅有赛季标注）：系统 MUST 使用可解释的时间归一规则，并在结果中标注「时间精度较低」；无法判定交集时 MUST 将该关系标为「无法判定」而非「不成立」
- **同一人两次输入 / 同一球员 ID**：MUST 提示无效输入，不展示自关系路径
- **外部履历数据源超时或限流**：MUST 展示可理解错误与重试；已缓存的履历可用于离线分析并标注数据新鲜度
- **大量短租借 / 青年队时段**：仍按时间交集规则计算；结果列表过长时 MUST 默认展示全部直接共同效力，间接路径仅展示最短一条
- **未登录用户**：MUST 要求登录后方可发起分析（与社区 Agent 类能力一致）；可浏览功能介绍但不返回完整分析
- **重复提交同一对球员**：短时间内重复请求 MUST 返回一致结论（可复用已有分析结果），避免重复施压外部数据源
- **权限**：普通登录用户仅可搜索与查看分析；不得访问内部采集任务控制面（若后续提供，仅管理员可见）

## Requirements *(mandatory)*

### Functional Requirements

#### 搜索与分析入口

- **FR-001**: 系统 MUST 提供球员关系搜索页，允许用户输入两名球员姓名并分别选择唯一球员身份
- **FR-002**: 关系分析页 URL MUST 携带决定该页的两名球员核心 ID，以支持刷新重入与面包屑导航
- **FR-003**: 用户 MUST 登录后方可提交关系分析；未登录时 MUST 引导登录

#### 履历数据

- **FR-004**: 系统 MUST 能为参与分析的球员持久化基础信息：姓名、出生日期、国籍、场上位置、当前效力球队（在数据源可提供时）
- **FR-005**: 系统 MUST 能为参与分析的球员持久化俱乐部职业履历条目：球队名称、加入时间、离队时间、转会类型、转会费（在数据源可提供时）
- **FR-006**: 系统 MUST 能为参与分析的球员持久化国家队效力条目（在数据源可提供时）：国家队名称、加入/离队或效力时间范围
- **FR-007**: 当本地缺少或过期履历时，系统 MUST 能从约定的外部公开履历数据源（Transfermarkt）按需获取并入库；采集失败时 MUST NOT 用虚构数据填补
- **FR-008**: 系统 MUST 记录履历数据的来源标识与最近成功同步时间，并在分析页可观察地展示数据新鲜度摘要

#### 直接关系判定

- **FR-009**: 系统 MUST 判定俱乐部队友关系：二人曾效力同一俱乐部，且效力时间区间存在交集；输出共同俱乐部列表与共同效力时间
- **FR-010**: 系统 MUST 判定国家队队友关系：二人曾代表同一国家队，且国家队效力时间存在交集；若国家队履历缺失，MUST 输出「无法判定」并说明原因
- **FR-011**: 时间交集判定 MUST 基于归一后的起止日期；边界规则（仅年份、缺省离队日等）MUST 在产品内一致，并在「无法判定」场景可解释

#### 转会与间接关系

- **FR-012**: 系统 MUST 分析并输出：是否存在直接转会关联（可验证的同一笔/直接关联转会叙事，以数据源字段为准）、是否先后加入同一俱乐部（时间上先后、无必须同时效力）
- **FR-013**: 系统 MUST 在球员与俱乐部构成的关系网络中计算两名球员之间的最短间接路径与关系距离；MUST 配置最大跳数上限，超出则视为无路径
- **FR-014**: 间接路径展示 MUST 明确节点类型（球员 / 俱乐部）；MUST NOT 在无数据支持时编造中间球星节点

#### 可视化

- **FR-015**: 系统 MUST 在关系分析页提供时间线视图，对齐展示两名球员与关系相关的效力时段，并突出共同效力区间（若存在）
- **FR-016**: 系统 MUST 在关系分析页提供关系图视图，展示与当前结论一致的球员节点、俱乐部节点及连接；路径与文字结论 MUST 一致
- **FR-017**: 加载、空结果、失败状态 MUST 有明确用户可见反馈，禁止静默空白

#### AI 预留（本 MVP 不交付）

- **FR-018**: 本 MVP MUST NOT 提供基于大语言模型的自然语言关系解读问答
- **FR-019**: 规格预留后续 AI 能力边界：未来 AI MUST 仅基于已入库的履历与本功能计算出的关系结论生成介绍；MUST NOT 捏造共同效力或比赛经历；超时/失败时 MUST 降级为结构化关系结论展示

### Key Entities

- **CareerPlayer（履历球员）**: 来自公开履历源的球员身份；含姓名、出生日期、国籍、位置、当前球队、外部源 ID、同步时间
- **ClubStint（俱乐部效力段）**: 球员在某俱乐部的一段效力；含加入/离队时间、转会类型、转会费（可选）
- **NationalTeamStint（国家队效力段）**: 球员代表某国家队的效力时间范围
- **PlayerPairAnalysis（球员对分析）**: 针对两名 CareerPlayer 的一次分析结果快照；含直接关系、转会关系、间接路径、距离、生成时间与数据新鲜度
- **RelationPath（关系路径）**: 有序节点列表（球员/俱乐部交替）及关系距离
- **TransferLink（转会关联摘要）**: 转会关系分析的结构化结论与依据引用

### 术语定义

| 术语 | 定义 |
|------|------|
| **俱乐部队友** | 二人曾效力同一俱乐部，且归一后的效力时间区间存在非空交集 |
| **国家队队友** | 二人曾代表同一国家队，且国家队效力时间区间存在非空交集 |
| **直接转会关联** | 数据源可支持的、与二人转会事件直接相关的可验证关联（非推断） |
| **先后加盟同一球队** | 二人曾先后（非必须同时）效力同一俱乐部 |
| **关系距离** | 间接路径上边的数量（跳数）；直接队友经共同俱乐部连通时距离按产品规则计为明确数值 |
| **无法判定** | 因履历缺失或时间精度不足，系统拒绝给出「成立/不成立」的二元结论 |
| **数据新鲜度** | 参与分析的履历最近一次成功同步时间相对当前的滞后说明 |

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001** [Deferrable: no]: 在本地已有两名目标球员完整俱乐部履历时，**95%** 的已知俱乐部队友样例对在提交后 **10 秒内** 返回正确的「曾经是队友」结论及至少一个正确的共同俱乐部名称
- **SC-002** [Deferrable: no]: 用户从搜索页完成「输入 → 消歧选择 → 提交」进入分析页的主路径，人工验收首次成功率 ≥ **90%**（不依赖开发者口头指导）
- **SC-003** [Deferrable: no]: 对约定测试集中「无直接队友、有间接连通」的球员对，**100%** 返回非空最短路径且路径节点均可在履历中验证；对「上限内不连通」的样例，**100%** 返回明确的无路径说明而非错误路径
- **SC-004** [Deferrable: no]: 关系分析页人工验收：时间线与关系图展示的共同俱乐部/路径与文字结论 **零矛盾**（抽检至少 5 对含关系、3 对无关系样例）
- **SC-005** [Deferrable: yes]: 外部履历源不可用时，使用未过期缓存完成分析的请求中，**100%** 在结果中展示数据新鲜度提示；无缓存时 **100%** 展示失败与重试，零虚构结论

## Out of Scope（本 Feature / MVP 不做）

- 基于大语言模型的自然语言问答与「为什么关系密切」生成文案（仅预留边界，见 FR-018/019）
- 预先爬取全站球员全集或商业级离线转会数据库导入
- 比赛级事件（共同出场、同场对决统计、共同夺冠叙事自动采信）—— MVP 不得将未经履历数据验证的荣誉叙事标为事实
- 与 Scout Agent（003）推荐对话的深度融合；本 feature 不修改 003 的 Player 契约（见 External Dependencies）
- 移动端原生 App、实时协作编辑、用户自定义关系标注
- 赔率、球探报告付费墙、视频集锦

## Assumptions

- 目标用户为社区已注册球迷；分析能力与 Agent 类功能同级，需登录
- MVP 数据采集为**按需获取**（用户选定球员时拉取），非全库预爬；允许后台维护适度缓存与过期刷新
- 间接关系网络以「球员—俱乐部—球员」二部图式连通为主；国家队节点可作为可选扩展，MVP 最短路径至少覆盖俱乐部连通
- 最大间接跳数默认上限为 **6**（可在 plan 中调参），超出即「无路径」
- 「共同获得三冠王」等荣誉叙事不作为 MVP 自动事实输出；时间线仅基于履历时段，不自动标注冠军条目
- 不修改 003 模块的 Player 模型与契约；若未来需要统一球员身份，须在 003 的 `tasks.md` 增补映射任务后再改对端
- 遵守数据源服务条款与合理访问频率；失败按 Edge Cases 处理
- 前台 E2E 以人工测试为准（项目约定），成功标准中的页面验收均为人工可验证条目
