# Harness 任务执行（Generator）

**上下文管理**: ✅ 保持当前上下文，但采用**轻量级上下文策略**

## 核心原则：依赖磁盘文件，不依赖对话记忆

```
每个任务的上下文来源：
1. spec文档（如 `[DATA_MODEL_DOC]`, `[API_CONTRACT_ROOT]`）→ 直接 Read 文件
2. 前置任务的产出 → Read 磁盘上已生成的代码文件，不是回忆对话历史
3. Generator模板 → 内化规则，不需要每次重新加载全文

❌ 错误：在上下文中保留前5个任务的完整对话过程
✅ 正确：每个任务只 Read 它直接依赖的已生成文件
```

## 指令

执行当前 Sprint 中的下一个（或指定的）任务。

### 输入参数

$ARGUMENTS — 可选，指定任务ID（如 "T009"）或 "batch"。不指定则自动取进度文件中的下一个未完成任务。

### 执行步骤

1. **定位任务**:
   - 找到最新的 `sprint-*-progress.md` 文件
   - 如果用户指定了任务ID，执行该任务
   - 否则找到第一个状态为 `[ ]` 的行（包括 🚧 批次门禁行）
   - **⛔ 批次门禁规则**: 如果下一个 `[ ]` 行是 `🚧 批次门禁`，**必须先执行 L1 Step 4 验证并标记通过，才能继续后续任务**。不得跳过门禁行去执行下一个普通任务。门禁行的执行方式见步骤 4 中 Step 4 的说明。

2. **加载最小上下文**（关键：只读必要文件，从磁盘读不从记忆读）:

   **判断任务类型，按类型加载：**

   | 任务类型 | 从 spec 文档读 | 从磁盘已生成文件读 |
   |---------|---------------|------------------|
   | 数据迁移 | `[DATA_MODEL_DOC]`（仅相关对象段落） | 无 |
   | 数据模型 | `[DATA_MODEL_DOC]`（仅相关对象段落） | 对应迁移/结构定义（确认字段与约束） |
   | 业务逻辑单元 | `[API_CONTRACT_ROOT]`（对应接口） + `spec.md`（对应验收场景） | 依赖的数据模型/业务接口（确认字段和方法签名） |
   | 接口入口 | `[API_CONTRACT_ROOT]`（对应接口） | 依赖的业务逻辑单元（确认方法签名） |
   | 用户界面接口适配 | `[API_CONTRACT_ROOT]`（对应接口） | 无（直接按契约生成） |
   | 用户界面页面/组件 | `spec.md`（对应验收场景） | 对应的用户界面接口适配（确认可用函数） |
   | 提示模板 | spec.md（相关场景描述） | 无 |

3. **执行 Generator（TDD 流程）**:

   根据 `.harness/prompts/generator.md` 中对应模板执行。
   **含业务逻辑的任务必须遵循 TDD**：

   ```
   Step 1 — Red:   先写测试，定义预期行为，确认测试失败
   Step 2 — Green: 写最小实现代码，使测试通过
   Step 3 — Refactor: 重构代码，保持测试通过
   ```

   不含业务逻辑的任务（数据迁移、配置、数据模型、传输对象）不要求 TDD。

4. **执行 Evaluator Level 1 — CI 门禁（强制，自动化）**:

   模拟 CI/CD 流水线，按顺序执行，**任一步失败则阻断，不得进入 L2**：

   ```
   ┌─────────────────────────────────────────────────────┐
   │ Step 1: 单元测试（门禁）                              │
   │                                                     │
   │ [TEST_COMMAND]                                      │
   │                                                     │
   │ ⛔ 阻断规则:                                         │
   │ - 任务含业务逻辑但无测试文件 → FAIL（未完成）           │
   │ - 任何测试失败 → FAIL                                │
   │ - 测试不通过不得进入下一步                             │
   ├─────────────────────────────────────────────────────┤
   │ Step 2: 编译                                        │
   │                                                     │
   │ [BUILD_COMMAND]                                     │
   │ [TYPECHECK_COMMAND]（如项目需要独立类型检查）           │
   ├─────────────────────────────────────────────────────┤
   │ Step 3: Lint                                        │
   │                                                     │
   │ [LINT_COMMAND] 无 ERROR                              │
   ├─────────────────────────────────────────────────────┤
   │ Step 4: 应用启动 + 集成验证（批次完成时执行）             │
   │                                                     │
   │ 4a. 启动项目依赖服务（如适用）→ 依赖健康                │
   │ 4b. [APP_START_COMMAND] → health UP + 数据迁移 OK     │
   │ 4c. [UI_START_COMMAND] → 页面可访问                   │
   │ 4d. 接口集成: 读取 [API_CONTRACT_ROOT] 构造真实请求     │
   │     对已实现的每个接口验证正常+异常路径                │
   │                                                     │
   │ ⛔ 编译通过 ≠ 能启动，启动成功 ≠ 功能可用              │
   │ 触发时机: 每个批次完成后 / Sprint Checkpoint           │
   └─────────────────────────────────────────────────────┘
   
   全部 ✅ → 进入 Level 2
   任一 ❌ → 进入 Corrector（最多3轮），3轮后仍失败则阻塞报告
   ```

   **🚧 批次门禁执行协议**:

   当定位到的下一个 `[ ]` 行是 `🚧 批次门禁` 时，执行以下流程：

   ```
   a. 按门禁行描述执行对应的 L1 Step 4 子步骤（4a/4b/4c/4d/4e/4f，按批次类型选择）
   b. 服务/核心批次: 必须执行 4a（依赖服务）+ 4b（[APP_START_COMMAND] + health + 数据迁移）
   c. 用户界面批次: 必须执行 4c（[UI_START_COMMAND] + 页面可访问）
   d. 含接口的批次: 必须额外执行 4d（契约驱动的真实请求验证）
   e. 用户界面批次（深度点击）: 必须执行 4e（Deep-walk · 从父页点进子页 · 检查子页接口）
   f. 涉真实外部服务 / 涉用户 UI 的批次: 必须执行 4f（[REAL_SERVICE_CHECK] 至少 1 次成功 + UI 截图 ≥ 2 张 + [MOCK_INDICATOR] 清洁）· 见 evaluator.md § 4f
   g. 验证通过 → 标记门禁行为 [x]，记录结果
   h. 验证失败 → 进入 Corrector，修正后重新验证
   i. 门禁通过后才能继续执行后续任务
   ```

   **👁 人工验证节点 HV 执行协议**（Sprint 47 血教训沉淀 · 2026-04-24）:

   当门禁行含 `👁 HV-M` 标记时：
   ```
   a. 执行 L1 Step 4 自动化部分（4a-4f 按批次类型）
   b. 自动化部分通过后 · **暂停**执行 · 输出给用户"HV-M 待签收"提示
      包含：当前 feature / 运行证据 URL · 预期验证动作 · 预计耗时
   c. 等用户反馈：
      - 用户回复"通过" → 标记门禁 [x] + 记录 "👁 HV-M PASS"
      - 用户回复"有问题"（附现象）→ 进入 Corrector · 走 /harness.fix
      - 用户超时未反馈 → 继续等 · 不自动通过
   d. 切勿执行代理自判 HV PASS · HV 的设计本意就是"自动执行之外的人给第二只眼"
   ```

   **进度文件中的门禁行格式**:
   ```
   - [ ] 🚧 **批次X.X门禁: L1 Step4 (描述)** | 结果: - |
   - [ ] 🚧 **批次X.X门禁: L1 Step4 + 👁 HV-M (~N min, 产品/用户)** | 结果: - |
   ```
   通过后更新为:
   ```
   - [x] 🚧 **批次X.X门禁: L1 Step4 (描述)** | 结果: ✅ health:UP migration:V{N} |
   - [x] 🚧 **批次X.X门禁: L1 Step4 + 👁 HV-M PASS** | 结果: ✅ 用户肉眼验收 + 截图 {N} 张 + runId={xxx} |
   ```

5. **执行 Evaluator Level 2**（按需，精准对照）:
   - 如果是接口入口任务: Read 对应 `[API_CONTRACT_ROOT]`，逐字段对照**刚写入磁盘的代码**
   - 如果是数据模型任务: Read `[DATA_MODEL_DOC]` 对应段落，逐字段对照
   - 输出简洁评分（4个维度各10分）
   - 总分 < 32 → 进入 Corrector
   - **评分表输出后无需保留在上下文中**——结果已记录到进度文件

5.5. **高风险任务即时触发 Level 4 Constitution 审查**:

   以下类型的任务完成后，**立即**执行 L4 审查（不等到 Sprint Checkpoint）：

   | 高风险任务类型 | 重点检查的 Constitution 原则 |
   |--------------|---------------------------|
   | 认证/权限入口 | VII 安全合规：认证检查、权限控制、审计日志 |
   | 高风险业务规则 | III 测试纪律：要求的分支覆盖 + VII 防篡改审计 |
   | 等级/权益变更 | III 测试纪律 + VII 审计日志 |
   | 外部服务调用实现 | IV 外部服务抽象层：接口调用、Prompt/配置外置、超时降级 |
   | 接口入口 | II 契约驱动：无契约外接口 + I 架构边界 |

   **判断规则**: 任务描述包含以下关键词之一 → 触发即时 L4：
   `认证、auth、安全、RBAC、权限、评分、scoring、等级变更、外部服务、[EXTERNAL_SERVICE_NAME]、接口入口`

   未命中关键词的任务 → L4 延迟到 Sprint Checkpoint 批量执行。

6. **Git Commit**（L1 + L2 + L4(如触发) 全部通过后才能提交）:
   - 先 `git add` 本任务生成/修改的文件（含测试文件），不要 `git add -A`
   - **如果项目根目录下存在 `.harness/scripts/ai-gen-commit.mjs` 文件，则必须使用水印脚本提交**（禁止 `git commit -m` 直接提交任务代码），否则不使用水印，并忽略下面与水印有关的约定，但提交的信息格式还是要严格遵守约定。
   - 水印提交命令：`node .harness/scripts/ai-gen-commit.mjs --commit "<提交信息>"`
   - 最终 message 格式：`<提交信息>[AI Gen_<校验码>]`
   - 示例：`[feat]:[CRMAIF_REQ_001][T021 新增导出][AI Gen_a1b2c3d4e5f6]`
   - 校验码仅绑定 message 正文（去掉 `[AI Gen_xxx]` 后缀后），禁止手写 `[AI Gen_xxx]`
   - 禁止 `--no-verify` 绕过 hook
   - Corrector 修正的代码合并在同一个 commit 中（不单独提交修正过程），仍走同一脚本
   - 提交信息拼装规则：
     - 类型简写：feat：新功能｜fix：bug / 问题修复｜docs：文档变更｜style：格式调整｜ref：代码重构｜test：测试代码｜chore：构建 / 工具改动｜tag：版本标记｜revert：回退提交｜perf：性能优化
     - feat/fix 固定模板：`[类型]:[单号][T{} 变更说明]`
       1. T{} 为关联的任务ID，有多个逗号隔开，接受 `-` 号，如：`T021-T022`
       2. 有单号填单号（REQ/ISSUE/BUG/CQM），无单填 `[]`，括号不能省
       - 例：`[feat]:[CRMAIF_REQ_001][T021 新增导出]`
       - 例：`[fix]:[][T022,T025,T029-T031 修复查询报错]`
     - 其余类型简写模板：`[类型]:[变更说明]`，可省略单号位
       - 例：`[docs]:[修订接口文档]`、`[perf]:[优化接口速度]`
   - 步骤 7 进度文件更新 **不用** 水印脚本，可普通 `git commit -m`

1. **更新进度文件**（持久化到磁盘）：
   - 标记任务为 `[x]` 并记录验证结果（含测试数量，如 `L1:✅(7/7)`）
   - 如有修正，记录到修正记录表
   - 更新度量快照
   - 进度文件的更新**不包含在任务 commit 中**，单独提交或累积提交

2. **清理上下文建议**:
   - 完成一个任务后，后续任务不需要回忆本次任务的细节
   - 如果连续执行多个任务，只保留当前任务的工作上下文
   - 当上下文开始压缩时，提示用户：
     "建议新开会话继续，进度已保存到 sprint-*-progress.md"

### 批量执行

如果用户传入 "batch"（如 `/harness.exec batch`），则连续执行当前批次内所有任务：
- 每个任务独立加载上下文（从磁盘读依赖文件）
- 并行标记 [P] 的任务使用子代理并行执行（每个子代理有干净上下文）
- 每个任务完成后立即更新进度文件
- 每个任务仍然独立做 L1+L2 验证
- 任何任务 FAIL 且3轮修正未解决 → 停止，报告阻塞
- 如果上下文接近极限 → 自动停止，提示新开会话继续

### 上下文容量监控

在连续执行过程中，如果出现以下信号，应主动建议用户新开会话：
- 对话轮次超过 20 轮
- 累计生成代码超过 2000 行
- 自动压缩了早期消息
- 用户反馈响应变慢

建议格式：
```
⚠️ 上下文建议：已连续执行 {N} 个任务，建议新开会话继续。
进度已保存：{已完成}/{总数} 任务完成
下一个任务：{任务ID} - {描述}
执行命令：/harness.start 或 /harness.exec {任务ID}
```
