# Metrics Prompt 模板 — 度量采集与报告

## 使用方式

在每个 Sprint 结束时（或每日）执行，生成度量报告。度量覆盖 Harness 五层的健康度和执行质量。

---

## 度量采集 Prompt

```
## Sprint 度量报告生成

请根据本 Sprint 的开发记录，生成以下度量数据。

### Sprint 信息
- Sprint编号: {N}
- Sprint目标: {目标}
- 时间范围: {起止日期}
- 计划任务数: {N}
- 实际完成任务数: {N}

---

### 一、约束层度量（Constraint Metrics）

衡量开发过程中约束规则的执行效果。

| 指标 | 公式 | 值 | 健康度 |
|------|------|------|--------|
| 硬约束违反次数 | 所有任务的 Level 3 硬约束 FAIL 累计 | {n} | 🟢=0, 🟡=1-2, 🔴≥3 |
| 软约束违反次数 | 所有任务的 Level 3 软约束 FAIL 累计 | {n} | 🟢≤2, 🟡=3-5, 🔴≥6 |
| 约束遵守率 | (总检查项 - 违反项) / 总检查项 × 100% | {n}% | 🟢≥95%, 🟡≥85%, 🔴<85% |
| Constitution 合规率 | 通过的原则数 / 7 × 100% (每个Checkpoint) | {n}% | 🟢=100%, 🟡≥85%, 🔴<85% |
| 未授权依赖引入数 | 引入了plan.md/constitution未列出的依赖 | {n} | 🟢=0, 🟡=1, 🔴≥2 |

---

### 二、信息层度量（Context Metrics）

衡量自动化辅助开发时获得的上下文信息的质量和覆盖度。

| 指标 | 公式 | 值 | 健康度 |
|------|------|------|--------|
| 契约覆盖率 | 有对应契约的接口数 / 总接口数 × 100% | {n}% | 🟢=100%, 🟡≥90%, 🔴<90% |
| 数据模型覆盖率 | 数据模型实现与 [DATA_MODEL_DOC] 一致的数 / 总数据模型数 × 100% | {n}% | 🟢=100%, 🟡≥90%, 🔴<90% |
| 验收场景覆盖率 | 有对应实现的验收场景数 / 总验收场景数 × 100% | {n}% | 🟢≥90%, 🟡≥70%, 🔴<70% |
| 上下文注入率 | 执行任务时成功注入完整上下文的次数 / 总任务数 × 100% | {n}% | 🟢≥95%, 🟡≥80%, 🔴<80% |
| Prompt模板使用率 | 使用Generator模板的任务数 / 总任务数 × 100% | {n}% | 🟢≥90%, 🟡≥70%, 🔴<70% |

---

### 三、验证层度量（Verification Metrics）

衡量质量门禁的有效性。

| 指标 | 公式 | 值 | 健康度 |
|------|------|------|--------|
| 首次通过率 | 第1轮Evaluator即PASS的任务数 / 总任务数 × 100% | {n}% | 🟢≥70%, 🟡≥50%, 🔴<50% |
| 平均验证分数 | 所有任务的Level 2总分平均值 / 50 | {n}/50 | 🟢≥40, 🟡≥35, 🔴<35 |
| 验证执行率 | 实际执行验证的任务数 / 应验证的任务数 × 100% | {n}% | 🟢=100%, 🟡≥90%, 🔴<90% |
| 构建通过率 | Level 1 构建检查通过次数 / 总检查次数 × 100% | {n}% | 🟢≥95%, 🟡≥80%, 🔴<80% |
| 测试通过率 | 通过的测试用例数 / 总测试用例数 × 100% | {n}% | 🟢≥95%, 🟡≥85%, 🔴<85% |
| E2E测试通过率 | [E2E_TOOL] 通过用例数 / 总E2E用例数 × 100% | {n}% | 🟢≥95%, 🟡≥80%, 🔴<80% |
| E2E场景覆盖率 | 有E2E测试的验收场景数 / 总验收场景数 × 100% | {n}% | 🟢≥90%, 🟡≥70%, 🔴<70% |
| E2E稳定性 | 3次重跑均通过的用例数 / 总E2E用例数 × 100% | {n}% | 🟢≥95%, 🟡≥85%, 🔴<85% |
| 契约一致性评分 | Level 2 契约检查的平均分 / 10 | {n}/10 | 🟢≥9, 🟡≥7, 🔴<7 |

---

### 四、修正层度量（Correction Metrics）

衡量修正效率和收敛性。

| 指标 | 公式 | 值 | 健康度 |
|------|------|------|--------|
| 需修正任务数 | Evaluator判定FAIL的任务数 | {n} | 仅记录 |
| 修正收敛率 | 3轮内修正成功的任务数 / 需修正任务数 × 100% | {n}% | 🟢≥90%, 🟡≥70%, 🔴<70% |
| 平均修正轮次 | 总修正轮次 / 需修正任务数 | {n} | 🟢≤1.5, 🟡≤2.5, 🔴>2.5 |
| 人工介入次数 | 超过3轮修正需要人工处理的次数 | {n} | 🟢=0, 🟡=1, 🔴≥2 |
| 回归引入率 | 修正过程中引入新问题的次数 / 总修正次数 × 100% | {n}% | 🟢≤5%, 🟡≤15%, 🔴>15% |
| 常见修正类型分布 | | | |
| - 契约不一致 | {n}次 ({n}%) | | |
| - 数据模型字段错误 | {n}次 ({n}%) | | |
| - 场景未覆盖 | {n}次 ({n}%) | | |
| - Constitution违反 | {n}次 ({n}%) | | |
| - 测试不通过 | {n}次 ({n}%) | | |

---

### 五、执行质量度量（Execution Quality Metrics）

衡量整体开发效率和产出质量。

| 指标 | 公式 | 值 | 健康度 |
|------|------|------|--------|
| Sprint完成率 | 完成任务数 / 计划任务数 × 100% | {n}% | 🟢≥90%, 🟡≥70%, 🔴<70% |
| 平均任务耗时 | 总开发时间 / 完成任务数 | {n}min | 仅记录，观察趋势 |
| 代码行数 | 本Sprint新增代码行数 | {n} | 仅记录 |
| 有效代码率 | (新增行数 - 修正时删除的行数) / 新增行数 × 100% | {n}% | 🟢≥85%, 🟡≥70%, 🔴<70% |
| 用户故事交付数 | 本Sprint完成的完整用户故事数 | {n} | 按计划 |
| Checkpoint通过率 | 通过的Checkpoint数 / 计划Checkpoint数 × 100% | {n}% | 🟢=100%, 🟡≥80%, 🔴<80% |
| 自动化辅助效率 | 实际耗时 / 纯人工预估耗时 | {n}x | 🟢≤0.4x, 🟡≤0.6x, 🔴>0.6x |

---

### 综合健康度仪表盘

| 层级 | 关键指标 | 状态 | 趋势(vs上Sprint) |
|------|---------|------|-----------------|
| 🛡️ 约束层 | 约束遵守率 {n}% | 🟢/🟡/🔴 | ↑/→/↓ |
| 📚 信息层 | 契约覆盖率 {n}% | 🟢/🟡/🔴 | ↑/→/↓ |
| ✅ 验证层 | 首次通过率 {n}% | 🟢/🟡/🔴 | ↑/→/↓ |
| 🔧 修正层 | 修正收敛率 {n}% | 🟢/🟡/🔴 | ↑/→/↓ |
| 📊 执行质量 | Sprint完成率 {n}% | 🟢/🟡/🔴 | ↑/→/↓ |

### 改进建议

基于度量数据，提出 Top 3 改进建议：

1. {针对最弱指标的改进措施}
2. {针对趋势下降的指标的改进措施}
3. {流程优化建议}
```
