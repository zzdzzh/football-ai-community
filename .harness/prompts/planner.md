# Planner Prompt 模板 — Sprint 规划

## 使用方式

将以下内容复制到 Code Agent 中执行。替换 `{变量}` 为实际值。

---

## Prompt

```
你是一个 Sprint 规划器（Planner）。请根据以下输入，将任务分解为可执行的 Sprint。

## 输入

### 任务清单
{粘贴 tasks.md 的内容，或指定路径让 Code Agent 读取}

### 约束条件
- 每个 Sprint 时长: {1周 / 2周}
- 团队规模: {N} 人
- 每人每Sprint可用工时: {X} 小时
- 自动化辅助开发效率系数: {倍率或 N/A}

### 规划规则

1. **依赖优先**: 严格遵守 tasks.md 中的 Phase 依赖顺序，不可跳过
2. **用户故事完整性**: 同一个用户故事的任务尽量放在同一个 Sprint，确保Sprint结束时有可验证的增量
3. **并行利用**: 标记 [P] 的任务可分配给不同成员并行执行
4. **验证节点**: 每个 Sprint 必须包含验证检查点（Checkpoint）
5. **MVP优先**: Sprint 1 必须瞄准最高优先级的 MVP 用户故事
6. **批次门禁**: 每个批次（Batch）末尾必须标注 L1 Step 4 门禁点。门禁点会体现在 progress 文件中作为强制检查行。

### 批次类型定义

每个批次（Batch）按内容分为三种类型，门禁验证内容不同：

| 批次类型 | 判断规则 | 门禁验证内容 |
|---------|---------|-------------|
| **服务/核心批次** | 包含数据迁移、数据模型、业务逻辑单元、接口入口任务 | `[APP_START_COMMAND]` 启动成功 + 数据迁移通过 + 已实现接口可调用 |
| **用户界面批次** | 包含用户界面页面/组件、接口适配、样式任务 | `[UI_START_COMMAND]` 启动成功 + `[TYPECHECK_COMMAND]` 通过（如适用）+ 页面可访问 |
| **集成批次** | 包含跨运行时联调、E2E、Checkpoint 任务 | `[APP_START_COMMAND]` + `[UI_START_COMMAND]` 启动成功 + 接口调用链成功 + `[E2E_TOOL]` 测试通过 |

混合批次（如同时包含服务/核心和用户界面任务）使用最高级别门禁（集成 > 服务/核心 > 用户界面）。

### 任务工时估算基准（自动化辅助开发）
- 数据迁移脚本: 0.5h
- 数据模型创建: 0.5h/个
- 业务逻辑单元实现: 1-2h
- 接口入口实现: 1h
- 契约测试: 0.5h
- 集成测试: 1h
- 用户界面接口适配: 0.5h
- 用户界面页面/组件（简单）: 1-2h
- 用户界面页面/组件（复杂，如富交互工作台）: 3-4h
- Seed 数据: 0.5h
- Prompt 模板: 0.5h

## 输出格式

对每个 Sprint 输出：

### Sprint {N}: {Sprint 目标}
**时间**: {起止日期}
**目标**: {一句话描述Sprint交付物}
**对应Phase**: {tasks.md 中的 Phase}

#### 任务清单
| 任务ID | 描述 | 负责人 | 预估工时 | 并行标记 |
|--------|------|--------|---------|---------|
| T001   | ...  | 成员A  | 0.5h    | [P]     |

#### 👁 人工验证节点（Sprint 47 血教训沉淀 · 2026-04-24 强制）

**规划阶段必须 identify · 至少 2 个节点 · 均涉用户可见 UI 或真实外部服务场景**。

规划目的：防"执行代理一路 autopilot 到 Sprint 收官才让用户验收"· 把问题暴露从"事后撤回判决"提前到"过程中及时 pivot"。

| 节点 ID | 所在批次 | 触发时机 | 验证动作 | 预期时间 | 验证人 |
|---|---|---|---|---|---|
| HV-1 | 批次 N（典型首个用户故事 MVP 完成）| 首个 User Story 可用时 | 浏览器打开关联页面 · 肉眼看核心 UI 路径 · 截图存证 | ~5 min | 产品 / 用户 |
| HV-2 | 批次 Polish（收官前）| 最后一个用户可见批次门禁 | `[REAL_SERVICE_CHECK]`（若涉真实外部服务）+ 关键 UI · 多条 US 路径各 1 次 | ~15-20 min | 产品 / 用户 |

**识别规则**:
- Feature 涉 `[EXTERNAL_SERVICE_NAME]` 真实服务场景 → 必须 ≥ 2 个 HV 节点
- Feature 有最终用户可见 UI（通用用户 / 运营人员 / 管理人员）→ 必须 ≥ 1 个 HV 节点（推荐 2）
- 纯内部重构（零 UI / 零真实外部服务）→ 可 0 个 HV 节点（显式说明）

**HV 节点在批次门禁行的标记格式**：
```
- [ ] 🚧 **批次N门禁: L1 Step4 + 👁 HV-M (~N min, {验证人})** | 结果: - |
```
通过后更新为：
```
- [x] 🚧 **批次N门禁: L1 Step4 + 👁 HV-M PASS** | 结果: ✅ 用户肉眼验收 + 截图 {N} 张 |
```

**HV 节点执行时**:
- 执行代理（Generator）**暂停**新任务 · 等待验证人反馈
- 验证人反馈"通过" → 继续批次
- 验证人反馈"有问题" → 走 Corrector 或 `/harness.fix` · 不继续后续任务
- 未标注 / 未执行 HV 直接收官 → Constitution VIII -2 · X -1

#### 验证检查点
- [ ] {自动化检查项1}
- [ ] {自动化检查项2}

#### Sprint 完成标准
- {可验证的交付物描述}
- **所有 HV 节点全部 PASS 签收**

#### 风险项
- {可能的阻塞或延期风险}

### Sprint 总览
| Sprint | 目标 | 任务数 | 总工时 | 关键交付物 |
|--------|------|--------|--------|-----------|
```

## 注意事项

- Sprint 规划产出保存到 `.harness/sprints/sprint-{n}.md`
- 每个 Sprint 开始前，用 Generator 模板逐个执行任务
- 每个 Sprint 结束后，用 Metrics 模板生成度量报告
