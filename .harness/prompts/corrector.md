# Corrector Prompt 模板 — 定向修正

## 使用方式

当 Evaluator 判定 FAIL 时，将失败项传入 Corrector。Corrector 进行最小化修正后，重新进入 Evaluator 验证。最多3轮修正，超过则人工介入。

---

## 修正 Prompt

```
## 定向修正（Corrector）

### 修正规则
1. **最小化修改**: 只修复 Evaluator 指出的具体问题，不重构无关代码
2. **不引入新问题**: 修正不能破坏已通过的检查项
3. **保持一致性**: 修正后的代码仍须符合契约、数据模型和 Constitution
4. **记录修正**: 每次修正说明改了什么、为什么

### 当前修正轮次: {第N轮 / 共3轮}

**进度文件记录格式**: 修正时在 progress 文件的修正记录表中追加行：
`| {任务ID} | {轮次}/3 | {原因} | {修正内容} |`
每轮修正更新轮次数（如 1/3 → 2/3 → 3/3），超过 3/3 标记为 `⚠️ BLOCKED`。

### Evaluator 失败项

{粘贴 Evaluator 报告中的失败项清单}

### 约束提醒

- 读取 `.specify/memory/constitution.md` 确保修正不违反任何原则
- 读取对应的 `[API_CONTRACT_ROOT]/[CONTRACT_FILE]` 确保修正后接口仍然一致
- 读取 `[DATA_MODEL_DOC]` 确保修正后数据模型仍然一致

### 请执行修正

对每个失败项：
1. 分析根因（为什么初次生成时出错）
2. 确定最小修复范围（改哪些文件的哪些行）
3. 执行修改
4. 说明修改内容

### 修正报告格式

| 失败项 | 根因 | 修复操作 | 影响文件 |
|--------|------|---------|---------|
| {问题} | {原因} | {做了什么} | {文件列表} |

修正完成后，请重新执行 Evaluator 的 Level {失败层级} 验证。
```

## 常见修正模式

### 模式1: 契约不一致

**症状**: 接口实现的字段名、类型或路径与接口契约定义不匹配

**修正方法**:
```
请对照 `[API_CONTRACT_ROOT]/[CONTRACT_FILE]` 中的以下端点或 operation：
- Path: {路径}
- Method: {方法}

当前实现中的不一致项：
{列出具体差异}

请修正接口入口和请求/响应对象，使其与契约完全一致。
```

### 模式2: 数据模型字段缺失或类型错误

**症状**: 数据模型实现与 [DATA_MODEL_DOC] 定义不一致

**修正方法**:
```
请对照 `[DATA_MODEL_DOC]` 中 {模型名} 的定义：

缺失字段: {字段列表}
类型错误: {字段: 预期类型 vs 实际类型}
约束缺失: {字段: 缺少的约束}

请修正数据模型实现，并确认对应的数据迁移或结构定义是否需要更新。
```

### 模式3: 验收场景未覆盖

**症状**: spec.md 中的 Given/When/Then 在代码中没有对应逻辑

**修正方法**:
```
以下验收场景未被覆盖：

场景: {Given... When... Then...}

请在 {业务逻辑单元/接口入口/用户界面页面} 中补充对应的逻辑。
说明这个场景在代码中的执行路径。
```

### 模式4: Constitution 违反

**症状**: 代码违反了 Constitution 中的硬约束

**修正方法**:
```
以下 Constitution 原则被违反：

原则 {编号}: {原则名}
违反描述: {具体描述}

请修正代码使其合规。如果确实需要违反软约束，请在 plan.md 的 Complexity Tracking 中记录理由。
```

### 模式5: 测试分支未覆盖

**症状**: 认证、权限或高风险业务规则未达到要求的分支覆盖

**修正方法**:
```
以下分支未被测试覆盖：

业务逻辑单元: {单元名}
方法: {方法名}
未覆盖分支:
- {分支描述1}
- {分支描述2}

请补充测试用例覆盖这些分支。
```

## 升级机制

| 修正轮次 | 操作 |
|---------|------|
| 第1轮 | 自动修正，重新验证 |
| 第2轮 | 自动修正，分析是否存在系统性问题 |
| 第3轮 | 最后一次自动修正，附带问题分析报告 |
| 第4轮+ | **人工介入**。输出完整的问题分析报告，包括：根因、已尝试的修正、建议的人工干预方式 |

```
## 人工介入请求
```
